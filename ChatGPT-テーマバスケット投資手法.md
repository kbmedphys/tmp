## 1. 参照すべき実務上の先行事例

もっとも近い実務例は、MSCI の thematic rotation 型インデックスです。MSCI World Thematic Rotation Factor Selection Index では、各テーマを MediaStats Megatrend Score でランクし、上位4テーマを選んで25%ずつ配分する設計が示されています。MediaStats Megatrend Score は、テーマごとのビジネス記述を使い、約10万のメディアソースから投資家の注目・センチメントの時系列およびクロスセクション変化を捉え、直近3か月のポジティブ・センチメント強度の変化を前3か月と比較して算出する、と説明されています。これは、今回の問題に対してかなり直接的なベンチマークになります。([MSCI](https://www.msci.com/documents/10199/f7ef115c-757e-55a1-bcda-2ca91f9c3068))

LSEG/MarketPsych も近い実務データです。LSEG は、MarketPsych Text Tagging Engine が非構造テキストを entity・topic・event・sentiment・emotion にタグ付けし、MarketPsych Analytics はニュースやソーシャルメディアからリアルタイムのセンチメント指標を作ると説明しています。また、Mentions、Sentiment、Optimism、Fear、Uncertainty、earnings forecast、fundamental sentiment、interest rates forecast などのスコアを扱うとされています。Refinitiv/LSEG 環境を使えるなら、この思想を自前テーマバスケットに拡張するのが自然です。([LSEG](https://www.lseg.com/en/data-analytics/market-data/quantitative-economic-data-solutions/marketpsych-analytics-and-models))

| 系統 | 使う目的 | 今回の使い方 |

近年のテーマ投資寄りの研究としては、THEME というフレームワークが参考になります。この研究は、テーマ投資はセクター境界をまたぎ、時間とともに関連銘柄が変化するため、静的 ETF 構成や専門家リストだけでは遅いと指摘しています。THEME は、テーマと構成銘柄の階層関係を使ってテキスト埋め込みを contrastive learning で調整し、さらにリターン時系列で temporal refinement を行う設計です。これは今回の「多様なテーマバスケットをニュースで動的に評価する」問題と非常に相性がよいです。([arXiv](https://arxiv.org/html/2508.16936v2))

## 3. 推奨する全体アーキテクチャ

結論として、実装すべきパイプラインは次です。

```text
テーマ定義
  ├─ テーマ名
  ├─ テーマ説明文
  ├─ 構成銘柄・ウェイト
  ├─ 各企業の事業概要
  ├─ TRBC/GICS/業種
  └─ バリュー/低PBR等の非ニュース属性
        ↓
テーマプロファイル生成
        ↓
Refinitiv ニュース取得
        ↓
ニュース前処理・entity linking・銘柄紐付け
        ↓
ニュース → テーマへの多ラベル帰属
        ↓
テーマ別ニュース特徴量
        ↓
盛り上がりスコア・センチメントスコア・新規性スコア
        ↓
価格・出来高・バリュエーション・リスク特徴量と統合
        ↓
テーマ選択・ポートフォリオ構築
        ↓
アウトオブサンプル検証
```

重要なのは、**テーマ帰属**と**投資判断**を分けることです。ニュースがあるテーマに関連するかどうかと、そのテーマを買うべきかどうかは別問題です。

## 4. テーマプロファイルの作り方

各テーマバスケット $k$ について、以下の情報を統合してテーマプロファイル $P_k$ を作ります。

$$
P_k =
\left[
\text{theme name},
\text{theme description},
\sum_{i \in k} w_{i,k} \cdot \text{company profile}_i,
\sum_{i \in k} w_{i,k} \cdot \text{industry tags}_i,
\text{style definition}
\right]
$$

たとえば「AI」テーマなら、テーマ名だけでなく、構成銘柄の事業概要から「semiconductor」「cloud infrastructure」「AI software」「data center」「GPU」「automation」などの表現を抽出します。「低PBR」テーマなら、ニュース本文との意味類似だけではなく、PBR、ROE、株主還元、自社株買い、政策保有株、東証要請などのキーワード・イベントを明示的に持たせるべきです。

テーマプロファイルは3層に分けるとよいです。

| 層 | 内容 | 目的 |
|---|---|---|
| Core | テーマの本質的説明 | 長期的な意味表現 |
| Constituents | 構成銘柄の事業概要をウェイト加重 | 実際の保有銘柄への紐付け |
| Catalysts | ニュース上で出やすい触媒語 | 短期ニュース検出 |

THEME 論文でも、テーマと既知構成銘柄の階層関係を教師信号にして、テーマと銘柄を同じ意味空間で近づける考え方が採られています。今回の実務実装では、まずは fine-tuning なしで SentenceTransformer/FinBERT 系の embedding を使い、データが蓄積したら contrastive learning に進むのが現実的です。([arXiv](https://arxiv.org/html/2508.16936v2))

## 5. ニュースをテーマに帰属させる方法

### 推奨は「多ラベル分類 + 類似度 + 企業ウェイト補正」

ニュース $d$ とテーマ $k$ の関連度を、以下のように分解します。

$$
\text{rel}_{d,k}
=
\lambda_1 \cdot \cos(e_d, e_{P_k})
+
\lambda_2 \cdot \sum_{i \in k} w_{i,k} \cdot \mathbf{1}(i \in d)
+
\lambda_3 \cdot \text{topic\_match}_{d,k}
+
\lambda_4 \cdot \text{keyword\_match}_{d,k}
$$

ここで、

| 項目 | 意味 |
|---|---|
| $\cos(e_d, e_{P_k})$ | ニュース本文とテーマプロファイルの埋め込み類似度 |
| $\sum w_{i,k} \mathbf{1}(i \in d)$ | ニュースに登場する企業がテーマ内でどれだけ重要か |
| $\text{topic\_match}$ | Refinitiv Topic とテーマ側 Topic の対応度 |
| $\text{keyword\_match}$ | 人手定義・LLM補助で作った触媒語との一致 |

この設計にする理由は、ニュースの出方がテーマごとに違うためです。AI や防衛のようなテーマはニュース本文との意味類似が効きやすい一方、低PBRや株主還元テーマは、企業名・バリュエーション・政策イベントの方が重要です。

テーマへの割当は単一ラベルではなく、**多ラベル**にすべきです。たとえば「NVIDIA の決算とデータセンター投資」は、AI、半導体、クラウド、電力インフラの複数テーマに関係します。MSCI の資料でも、テーマ間で記述語が重なり、記事が複数テーマに共有される可能性があることが明記されています。([MSCI](https://www.msci.com/documents/10199/f7ef115c-757e-55a1-bcda-2ca91f9c3068))

## 6. BERTopic / Dynamic Topic Model の使い方

Refinitiv Topic が粗い場合、ニュース集合から自前の潜在トピックを抽出します。BERTopic は、Transformer で文書 embedding を作り、それをクラスタリングし、c-TF-IDF でトピック表現を作る手法です。短いニュース見出し・記事要約を扱う実務では、古典的 LDA より扱いやすいことが多いです。([arXiv](https://arxiv.org/abs/2203.05794))

一方、テーマの時間変化を追いたい場合は Dynamic Topic Model が理論的に自然です。Blei and Lafferty の Dynamic Topic Models は、時系列上でトピックがどう変化するかを分析するための確率モデルとして提案されています。ニュース上の「AI」の語彙が、半導体→生成AI→データセンター→電力→ロボティクスへ変わるような変化を追う用途に適しています。([mimno.infosci.cornell.edu](https://mimno.infosci.cornell.edu/info6150/readings/dynamic_topic_models.pdf?utm_source=chatgpt.com))

実務実装では、最初から DTM を重く実装するより、次の順番がよいです。

| 段階 | 手法 | 用途 |
|---|---|---|
| Phase 1 | テーマ説明 embedding × ニュース embedding | 既存テーマへのニュース帰属 |
| Phase 2 | BERTopic | 未分類ニュースから新興トピック発見 |
| Phase 3 | 時系列 BERTopic / Dynamic Topic Model | テーマ語彙・関心の時間変化 |
| Phase 4 | contrastive fine-tuning | 自社テーマバスケット専用モデル化 |

## 7. テーマの「盛り上がり」をどう定義するか

単純なニュース件数では不十分です。大型テーマや大型株テーマは常に記事が多く、ニッチテーマは少数記事でもインパクトが大きいからです。MSCI/MediaStats 型の考え方に倣い、**水準ではなく変化率・標準化・ポジティブ強度の変化**を見るべきです。MSCI の資料でも、直近3か月の positive sentiment intensity change を前3か月と比較する設計が示されています。([MSCI](https://www.msci.com/documents/10199/f7ef115c-757e-55a1-bcda-2ca91f9c3068))

推奨するテーマ盛り上がりスコアは以下です。

$$
\text{Buzz}_{k,t}
=
z\left(
\log(1 + \text{mentions}_{k,t})
-
\log(1 + \text{mentions}_{k,t-L:t-1}^{avg})
\right)
$$

$$
\text{Tone}_{k,t}
=
z\left(
\frac{
\sum_{d \in t} \text{rel}_{d,k} \cdot \text{sentiment}_{d}
}{
\sum_{d \in t} \text{rel}_{d,k} + \epsilon
}
\right)
$$

$$
\text{Novelty}_{k,t}
=
1 -
\max_{\tau < t}
\cos(e_{news,k,t}, e_{news,k,\tau})
$$

$$
\text{ThemeHeat}_{k,t}
=
a \cdot \text{Buzz}_{k,t}
+
b \cdot \text{Tone}_{k,t}
+
c \cdot \text{Novelty}_{k,t}
+
d \cdot \text{Breadth}_{k,t}
-
e \cdot \text{Crowding}_{k,t}
$$

各要素の意味は次の通りです。

| 指標 | 解釈 | 投資上の意味 |
|---|---|---|
| Buzz | ニュース量の平常時対比 | 注目度の上昇 |
| Tone | ポジティブ/ネガティブ方向 | 買い/売りの方向性 |
| Novelty | 過去にない新しい話題か | 新規材料・テーマ転換 |
| Breadth | 関連銘柄の何割にニュースが広がったか | テーマ全体への波及 |
| Crowding | 価格・出来高・ニュース過熱の合成 | 逆張り警戒 |

ここで特に重要なのは **Breadth** です。テーマ内の1銘柄だけが盛り上がっているのか、バスケット全体にニュースが広がっているのかで、テーマ投資としての意味が大きく異なります。

$$
\text{Breadth}_{k,t}
=
\sum_{i \in k} w_{i,k} \cdot \mathbf{1}(\text{news\_score}_{i,t} > q)
$$

たとえば AI テーマで NVIDIA だけが強い場合と、半導体、クラウド、電力、ソフトウェア、自動化関連まで広がっている場合では、後者の方がテーマとしての持続性が高いと判断できます。

## 8. 投資シグナルへの変換

ニュース由来の ThemeHeat をそのまま買うのではなく、価格・バリュエーション・流動性・リスクと統合します。

$$
\text{Score}_{k,t}
=
\beta_1 \cdot \text{ThemeHeat}_{k,t}
+
\beta_2 \cdot \text{PriceMomentum}_{k,t}
+
\beta_3 \cdot \text{EarningsRevision}_{k,t}
+
\beta_4 \cdot \text{ValuationAttractiveness}_{k,t}
-
\beta_5 \cdot \text{Volatility}_{k,t}
-
\beta_6 \cdot \text{Crowding}_{k,t}
$$

テーマ投資では、ニュースの盛り上がりがポジティブでも、すでに価格に織り込まれている可能性があります。したがって、最低限、以下の4つは分けて検証すべきです。

| シグナル | 役割 |
|---|---|
| News-only | ニュースだけに予測力があるか |
| Price momentum-only | 価格モメンタムの代替にすぎないか |
| News + Momentum | ニュースがモメンタムを補完するか |
| News surprise | 価格がまだ反応していないニュースを拾えるか |

MSCI の methodology でも、MediaStats データが利用できない場合や stale な場合、3か月・6か月のリスク調整後価格モメンタムをフォールバックとして使う設計が示されています。これは、ニュースシグナルと価格モメンタムを分けて扱うべきことを示す実務上の参考になります。([MSCI](https://www.msci.com/documents/10199/f7ef115c-757e-55a1-bcda-2ca91f9c3068))

## 9. ポートフォリオ構築

最初の実装はシンプルでよいです。

### 案A：上位Nテーマを等ウェイト

MSCI 型に近いです。

$$
w_{k,t}
=
\begin{cases}
1/N & \text{if } k \in \text{TopN}(\text{Score}_{k,t}) \\
0 & \text{otherwise}
\end{cases}
$$

メリットは、解釈しやすく、過剰最適化しにくいことです。MSCI のテーマローテーション型インデックスも、上位4テーマを選んで25%ずつ配分する考え方を採っています。([MSCI](https://www.msci.com/documents/10199/f7ef115c-757e-55a1-bcda-2ca91f9c3068))

### 案B：スコア比例ウェイト

$$
w_{k,t}
=
\frac{\max(\text{Score}_{k,t},0)}
{\sum_j \max(\text{Score}_{j,t},0)}
$$

テーマ数が多く、強弱を滑らかに反映したい場合に使います。

### 案C：リスク調整スコア

$$
\text{RiskAdjScore}_{k,t}
=
\frac{\text{Score}_{k,t}}{\hat{\sigma}_{k,t}}
$$

テーマごとのボラティリティ差が大きい場合はこちらが適しています。

### 案D：ニュース・モメンタム二段階

実務ではこれが一番堅いです。

1. News Score で候補テーマを絞る  
2. 候補内で価格モメンタム、バリュエーション、リスクで順位付け  
3. 上位テーマに配分  
4. テーマ内は既存バスケットウェイト、または流動性・リスクで補正  

## 10. 低PBRなど「ニューステーマでないテーマ」の扱い

低PBR、株主還元、高配当、クオリティ、赤字成長株、地銀のようなテーマは、AI や防衛と違い、ニュース本文の意味類似だけでは弱いです。

この場合は、テーマを次のように分解します。

| テーマ型 | 主特徴量 | ニュース側の見るべきイベント |
|---|---|---|
| 産業テーマ | 意味類似、構成銘柄、Topic | 技術革新、規制、需要、設備投資 |
| マクロテーマ | マクロ Topic、国・金利・為替 | 中銀、金利、財政、資源価格 |
| スタイルテーマ | 財務指標、価格、ファクター | 自社株買い、増配、ROE改善、資本政策 |
| 政策テーマ | 政策文書、法案、予算 | 補助金、規制緩和、国策 |
| イベントテーマ | M&A、決算、訴訟、承認 | 個別イベントの波及 |

低PBRテーマなら、ニュース上の「東証要請」「資本効率」「自社株買い」「増配」「政策保有株売却」「PBR 1倍割れ」「アクティビスト」などを catalyst として明示的に定義し、PBR や ROE などの構造データと統合するべきです。

## 11. 検証設計

この手法は、必ずアウトオブサンプルで検証すべきです。Glasserman らの研究が示すように、ニューストピックをリターン説明に使う場合、教師ありでリターンに合わせすぎると過学習しやすいです。([SSRN](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=3705738))

最低限の検証は次です。

| 検証 | 内容 |
|---|---|
| 帰属精度 | ニュースが正しいテーマに割り当たっているかを人手ラベルで確認 |
| IC | ThemeHeat と将来リターンの Rank IC |
| 分位ポートフォリオ | News Score 上位 vs 下位のリターン差 |
| アブレーション | Buzz、Tone、Novelty、Breadth の寄与分解 |
| 価格モメンタム対比 | ニュースが価格モメンタム以上の情報を持つか |
| セクター中立 | セクター偏りを除いても有効か |
| イベント日除外 | 決算・大型イベントだけに依存していないか |
| 取引コスト後 | リバランス頻度と turnover を考慮 |
| ライブ風検証 | ニュース配信時刻・利用可能時刻を厳密に反映 |

特に重要なのは、**テーマ帰属モデルの精度**と**投資シグナルの有効性**を分けて評価することです。帰属精度が高くてもリターン予測力があるとは限りません。

## 12. 実装優先順位

最初から LLM fine-tuning や複雑な DTM に進むより、以下の順番が実務的です。

### Phase 1：堅牢なベースライン

- テーマプロファイルを作成  
- ニュース本文・見出しを embedding  
- テーマとの cosine similarity で多ラベル付与  
- 企業名/entity linking で補正  
- Buzz、Tone、Breadth を日次/週次で作成  
- 上位テーマ等ウェイトでバックテスト  

### Phase 2：Refinitiv/MarketPsych 的な特徴量拡張

- Mentions
- Sentiment
- Optimism
- Fear
- Uncertainty
- Event type
- Topic intensity
- Company-level aggregation
- Theme-level aggregation

LSEG/MarketPsych は、ニュースやソーシャルメディアを構造化スコアに変換し、sentiment、emotion、topic、event を使う枠組みを提供しているため、この設計の参考になります。([LSEG](https://www.lseg.com/en/data-analytics/market-data/quantitative-economic-data-solutions/marketpsych-analytics-and-models))

### Phase 3：未分類ニュースのクラスタリング

- 類似度が低いニュースを BERTopic に投入
- 新興トピックを抽出
- 既存テーマに追加するか、新規テーマ候補にする
- 週次/月次で人手レビュー

BERTopic は Transformer embedding、クラスタリング、c-TF-IDF によるトピック表現を組み合わせるため、ニュース見出しや短文の潜在テーマ抽出に使いやすいです。([arXiv](https://arxiv.org/abs/2203.05794))

### Phase 4：テーマ専用モデル化

- テーマと構成銘柄を正例
- 近いが別テーマの銘柄を hard negative
- ニュースで共起する銘柄・テーマを補助教師
- contrastive learning でテーマ専用 embedding を作成
- リターンや出来高で temporal refinement

THEME 論文のように、テーマ・構成銘柄・テキスト・リターンを同時に使うと、一般的な embedding よりテーマ投資に特化した表現を作れる可能性があります。([arXiv](https://arxiv.org/html/2508.16936v2))

## 13. 最終的な推奨手法

今回の条件では、私は次の設計を推奨します。

**中核モデル：**

$$
\text{ThemeHeat}_{k,t}
=
0.30 \cdot \text{Buzz}_{k,t}
+
0.25 \cdot \text{Tone}_{k,t}
+
0.20 \cdot \text{Breadth}_{k,t}
+
0.15 \cdot \text{Novelty}_{k,t}
-
0.10 \cdot \text{Crowding}_{k,t}
$$

**最終スコア：**

$$
\text{FinalScore}_{k,t}
=
0.50 \cdot \text{ThemeHeat}_{k,t}
+
0.30 \cdot \text{RiskAdjustedMomentum}_{k,t}
+
0.20 \cdot \text{FundamentalSupport}_{k,t}
$$

**投資ルール：**

| 項目 | 推奨 |
|---|---|
| リバランス | 週次または月次 |
| ニュース集計 | 日次→週次平滑化 |
| テーマ選択 | 上位3〜5テーマ |
| 配分 | 等ウェイトまたはリスク調整等ウェイト |
| テーマ内配分 | 既存バスケットウェイトを基本 |
| リスク管理 | ボラターゲット、最大ウェイト、セクター偏り制約 |
| 検証 | Rank IC、分位、ロングオンリー、コスト後、価格モメンタム対比 |

まずは **「embedding 多ラベル帰属 + Buzz/Tone/Breadth + 価格モメンタム補完」** で十分です。その後、BERTopic で未分類ニュースを拾い、最終的に contrastive learning でテーマ専用 embedding に進むのが、学術的にも実務的にも一番筋がよいです。

## 2.3 MSCI Thematic Rotation / MediaStats Megatrend Score

**位置付け：テーマローテーションの実務ベンチマーク**

MSCI World Thematic Rotation Select Index は、複数の MSCI テーマ指数から上位4テーマを毎月選択するインデックスです。選択は MKT MediaStats が算出する theme-specific media sentiment score、すなわち MediaStats Megatrend Score に基づくと説明されています。([MSCI](https://www.msci.com/indexes/index/752499?utm_source=chatgpt.com))

MSCI の methodology では、MediaStats Megatrend Score は各 thematic index について毎月算出され、各テーマ指数の business description を用いて、メディア記事から investor attention と sentiment の時系列・横断面変化を捉えると説明されています。([MSCI](https://www.msci.com/eqb/methodology/meth_docs/MSCI_World_Thematic_Rotation_Select_Index_Methodology.pdf?utm_source=chatgpt.com))

**我々への示唆**

これは今回の問題に最も近い実務事例です。特に重要なのは以下です。

| MSCI の考え方 | 我々の実装 |
|---|---|
| テーマごとの business description を使う | テーマプロファイルを作る |
| メディアから attention と sentiment を測る | Buzz と Tone を作る |
| テーマ横断でスコア比較 | 全テーマを z-score 化して比較 |
| 月次で上位テーマを選ぶ | 週次または月次で上位テーマを選ぶ |
| 価格モメンタムを補助的に使う | ニュース + モメンタムの合成スコアにする |

MSCI 型をそのまま模倣するなら、最初のベースラインは「テーマ別ニュースセンチメント変化率で上位テーマを選ぶ」だけでも成立します。

## 2.4 Business News and Business Cycles

**位置付け：ニュース注目度を時系列マクロ・市場情報に変換する学術的根拠**

Bybee, Kelly, Manela, Xiu の研究では、約80万件の Wall Street Journal 記事からトピックモデルを推定し、ニュースを解釈可能な topical themes に要約し、各テーマへのニュース注目配分を時系列で数量化しています。そのニュース注目は経済活動をよく追跡し、株式市場リターンの予測にも関係すると報告されています。([NBER](https://www.nber.org/papers/w29344?utm_source=chatgpt.com))

**我々への示唆**

ニュースを個別記事のまま使うのではなく、次のように変換するべきです。

```text
記事単位ニュース
  ↓
トピック・テーマ別の注目配分
  ↓
テーマ別ニュース時系列
  ↓
将来リターン・リスクとの関係を検証
```

つまり、目的は「記事を読むこと」ではなく、**テーマごとの attention share / sentiment share を時系列特徴量にすること**です。

---

## 2.5 Sentence-BERT / Embedding 類似度

**位置付け：テーマ説明とニュース本文の意味的対応付け**

Sentence-BERT は、BERT を siamese / triplet network 構造で修正し、文や文章を cosine similarity で比較可能な意味ベクトルに変換する手法です。元論文では、通常の BERT は semantic similarity search や clustering にそのまま使いにくいため、SBERT により意味的に近い文をベクトル空間上で近く配置できるようにしたと説明されています。([ACLアントロポロジー](https://aclanthology.org/D19-1410/?utm_source=chatgpt.com))

**我々への示唆**

テーマ名だけでニュースを拾うと漏れが大きくなります。たとえば「AI」テーマでも、ニュース本文には “GPU demand”, “data center capex”, “inference workload”, “cloud infrastructure” などが出ます。したがって、テーマ説明・構成銘柄概要・ニュース本文を同じ embedding 空間で比較する必要があります。

---

## 2.6 BERTopic

**位置付け：既存テーマに入らない新興ニュース群の発見**

BERTopic は、pre-trained transformer で文書 embedding を作り、クラスタリングした後、class-based TF-IDF でトピック表現を生成する topic model です。論文では、document embedding、クラスタリング、c-TF-IDF によって coherent topic representation を作ると説明されています。([arXiv](https://arxiv.org/abs/2203.05794?utm_source=chatgpt.com))

**我々への示唆**

BERTopic は、テーマバスケットへの直接帰属よりも、次の用途に向いています。

- 既存テーマにうまく入らないニュースをクラスタリングする。
- 新しいテーマ候補を発見する。
- 既存テーマのニュース語彙が変化しているか確認する。
- 「AI → データセンター電力 → 電線・変圧器」のようなテーマ拡張を検知する。

---

## 2.7 Dynamic Topic Models

**位置付け：テーマ語彙・話題構造の時間変化の把握**

Blei and Lafferty の Dynamic Topic Models は、大規模文書集合におけるトピックの時間発展を分析する確率的時系列モデルです。論文では、topics の time evolution を分析するための probabilistic time series models と説明されています。([mimno.infosci.cornell.edu](https://mimno.infosci.cornell.edu/info6150/readings/dynamic_topic_models.pdf?utm_source=chatgpt.com))

**我々への示唆**

最初から DTM を実装する必要はありません。ただし、テーマの盛り上がりが一時的なのか、語彙が持続的に変化しているのかを見たい場合には有用です。

実務上は、まずは rolling BERTopic / monthly topic snapshot で代替し、必要になったら DTM に進むのがよいです。

---

## 2.8 FinBERT

**位置付け：金融ニュースの tone 判定**

FinBERT は、金融文脈の sentiment analysis に特化した BERT 系モデルです。元論文では、金融文脈では一般的な言語モデルでは不十分であり、金融ドメインで事前学習・調整したモデルが有効だと説明されています。([arXiv](https://arxiv.org/abs/1908.10063?utm_source=chatgpt.com))

**我々への示唆**

テーマスコアに使う tone は、一般的な positive/negative 辞書よりも、金融文脈に強いモデルを使うべきです。

たとえば以下のような判定が重要です。

| ニュース表現 | 一般辞書 | 金融的解釈 |
|---|---:|---:|
| “inflation falls” | negative word なし | 金利低下期待で株式にポジティブ |
| “guidance cut” | cut は中立になり得る | 業績にネガティブ |
| “buyback expansion” | expansion は一般に positive | 資本政策テーマに強いポジティブ |
| “regulatory investigation” | investigation | セクターにネガティブ |

## 2.9 Glasserman et al. “Choosing News Topics to Explain Stock Market Returns”

**位置付け：過学習への警告**

Glasserman らは、ニューストピックで株式リターンを説明する手法を比較し、supervised LDA はリターンに過剰適合しやすく、アウトオブサンプル性能を損なう可能性があると報告しています。([SSRN](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=3705738&utm_source=chatgpt.com))

**我々への示唆**

重要なのは、**リターンに合うニュース分類を後付けで作りすぎないこと**です。

したがって、最初の設計では以下を避けるべきです。

- リターンを教師にしてニュース分類モデルを直接最適化する。
- テーマ定義をバックテスト結果に合わせて頻繁に書き換える。
- ニューススコアの重みを過剰に最適化する。
- テーマ数が多い状態で unconstrained な機械学習ランキングにする。

まずは、**事前定義されたテーマプロファイル + ニュース特徴量 + シンプルなスコア合成**で始めるべきです。

# 3. 我々が取るべき手法

## 結論：4層モデルにする

我々のテーマ投資モデルは、以下の4層に分けるべきです。

```text

```

# 4. Layer 1：lseg.data で取得するデータ

## 4.1 ニュースデータ

# 5. Layer 2：ニュースをテーマに帰属させる

ここが最重要です。Refinitiv Topic は使いますが、それだけでテーマバスケットと対応させてはいけません。

## 5.1 テーマプロファイルを作る

各テーマ $k$ について、以下を統合したテキストを作ります。

$$
P_k =
[
\text{theme name},
\text{theme description},
\sum_i w_{i,k} \cdot \text{company profile}_i,
\sum_i w_{i,k} \cdot \text{TRBC/GICS tags}_i,
\text{catalyst keywords}
]
$$

例：AI テーマ

```yaml
theme_id: ai_infrastructure
theme_name: AI infrastructure
core_description: >
  Companies exposed to artificial intelligence compute, GPU, accelerator,
  data center, cloud infrastructure, inference, machine learning software.
catalysts:
  - GPU demand
  - data center capex
  - AI server
  - inference workload
  - cloud AI
  - semiconductor supply
  - hyperscaler spending
```

例：低PBRテーマ

```yaml
theme_id: low_pbr_reform
theme_name: Low PBR / capital efficiency reform
core_description: >
  Companies trading below book value or targeted by capital efficiency reforms,
  buybacks, dividend increases, governance reform, activist pressure.
catalysts:
  - share buyback
  - dividend hike
  - capital efficiency
  - PBR below one
  - governance reform
  - activist investor
  - cross-shareholding reduction
```

## 5.2 ニュース関連度スコア

ニュース $d$ とテーマ $k$ の関連度を次で定義します。

$$
\text{Rel}_{d,k}
=
\lambda_1 \cdot \cos(e_d, e_{P_k})
+
\lambda_2 \cdot \text{EntityOverlap}_{d,k}
+
\lambda_3 \cdot \text{TopicMatch}_{d,k}
+
\lambda_4 \cdot \text{KeywordMatch}_{d,k}
$$

各項目は以下です。

| 項目 | 内容 |
|---|---|
| embedding similarity | ニュース本文とテーマプロファイルの意味類似 |
| EntityOverlap | ニュースに登場する RIC がテーマ構成銘柄に含まれるか |
| TopicMatch | Refinitiv Topic とテーマの対応 |
| KeywordMatch | 触媒語との一致 |

初期値は以下でよいです。

$$
\lambda_1=0.45,\quad
\lambda_2=0.30,\quad
\lambda_3=0.15,\quad
\lambda_4=0.10
$$

ただし、テーマ型に応じて変えます。

| テーマ型 | embedding | entity | topic | keyword |
|---|---:|---:|---:|---:|
| AI・防衛・電力インフラ | 高 | 中 | 中 | 中 |
| 低PBR・高配当・株主還元 | 中 | 高 | 低 | 高 |
| 地銀・半導体・医薬品 | 中 | 高 | 中 | 中 |
| マクロテーマ | 中 | 低 | 高 | 高 |

# 6. Layer 3：テーマ別ニュース特徴量を作る

ニュースをテーマに割り当てた後、日次または週次で以下を作ります。

## 6.1 Buzz：注目度

$$
\text{Buzz}_{k,t}
=
z\left(
\log(1+\text{Mentions}_{k,t})
-
\log(1+\overline{\text{Mentions}}_{k,t-L:t-1})
\right)
$$

これはテーマの平常時対比の盛り上がりです。

## 6.2 Tone：ニュース方向性

$$
\text{Tone}_{k,t}
=
z\left(
\frac{
\sum_{d \in t} \text{Rel}_{d,k} \cdot \text{Sentiment}_d
}{
\sum_{d \in t} \text{Rel}_{d,k}+\epsilon
}
\right)
$$

Sentiment は FinBERT、LSEG/MarketPsych 系スコア、または独自分類で作ります。

## 6.3 Breadth：テーマ内の広がり

$$
\text{Breadth}_{k,t}
=
\sum_{i \in k} w_{i,k}
\cdot
\mathbf{1}(\text{NewsScore}_{i,t} > q)
$$

これは非常に重要です。テーマ内の1銘柄だけが盛り上がっているのか、テーマ全体に広がっているのかを区別できます。

## 6.4 Novelty：新規性

$$
\text{Novelty}_{k,t}
=
1 -
\max_{\tau<t}
\cos(e_{k,t}^{news}, e_{k,\tau}^{news})
$$

過去と似たニュースの繰り返しか、新しい材料かを測ります。

## 6.5 Crowding：過熱・織り込み

$$
\text{Crowding}_{k,t}
=
z(\text{Turnover}_{k,t})
+
z(\text{ShortTermMomentum}_{k,t})
+
z(\text{NewsBuzz}_{k,t})
+
z(\text{ValuationRichness}_{k,t})
$$

ニュースが良くても、すでに価格・出来高が過熱している場合は減点します。

---

# 7. Layer 4：投資スコア化

## 7.1 ThemeHeat

まずニュース由来のテーマ熱量を作ります。

$$
\text{ThemeHeat}_{k,t}
=
0.30 \cdot \text{Buzz}_{k,t}
+
0.25 \cdot \text{Tone}_{k,t}
+
0.20 \cdot \text{Breadth}_{k,t}
+
0.15 \cdot \text{Novelty}_{k,t}
-
0.10 \cdot \text{Crowding}_{k,t}
$$

## 7.2 最終投資スコア

ニュースだけで投資判断せず、価格・ファンダメンタルズを統合します。

$$
\text{FinalScore}_{k,t}
=
0.50 \cdot \text{ThemeHeat}_{k,t}
+
0.30 \cdot \text{RiskAdjustedMomentum}_{k,t}
+
0.20 \cdot \text{FundamentalSupport}_{k,t}
$$

### 各要素

| 要素 | 内容 |
|---|---|
| ThemeHeat | ニュース由来の盛り上がり |
| RiskAdjustedMomentum | 1〜3か月リターン / ボラ |
| FundamentalSupport | 低PBR、ROE改善、EPS revision、業績修正など |
| Crowding penalty | 過熱・短期反転リスク |

## 推奨ルール：ニュース主導のテーマローテーション

| 項目 | 推奨 |
|---|---|
| 集計頻度 | 日次でニュース集計、週次または月次でリバランス |

# 9. テーマ型ごとの処理分岐

テーマが多様なので、全テーマを同じロジックで扱うと精度が落ちます。以下の分類を入れるべきです。

| テーマ型 | 例 | 主な情報源 | 重視する特徴 |
|---|---|---|---|
| 産業テーマ | AI、防衛、電力インフラ、半導体 | ニュース本文、企業概要 | embedding, Buzz, Breadth |
| 政策テーマ | 国土強靭化、防衛、GX | Topic, 政策ニュース | Topic, Keyword, Event |
| スタイルテーマ | 低PBR、高配当、小型成長 | 財務・価格・資本政策ニュース | entity, keyword, valuation |
| マクロテーマ | 円高メリット、金利上昇、資源 | マクロニュース | Topic, macro variables |
| イベントテーマ | 自社株買い、M&A、決算 | イベントニュース | event extraction, sentiment |

特に低PBRテーマは、単なるニュース embedding では弱いです。以下のように、構造データとニュースを混ぜる必要があります。

$$
\text{FundamentalSupport}_{k,t}
=
z(-\text{PBR}_{k,t})
+
z(\Delta \text{ROE}_{k,t})
+
z(\text{BuybackYield}_{k,t})
+
z(\text{DividendRevision}_{k,t})
$$

# 10. 検証設計

ニュースモデルは過学習しやすいので、Glasserman らの警告を踏まえ、リターンに合わせてニュース分類を最適化しすぎない設計にします。([SSRN](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=3705738&utm_source=chatgpt.com))

最低限、以下を検証します。

| 検証 | 内容 |
|---|---|
| テーマ帰属精度 | 人手ラベル 200〜500件で precision/recall を確認 |
| Rank IC | ThemeHeat と翌週/翌月リターンの順位相関 |
| 分位ポートフォリオ | ニューススコア上位 vs 下位 |
| News-only | ニュースだけで有効か |
| Momentum-only | 価格モメンタムだけとの比較 |
| News + Momentum | 補完効果があるか |
| Breadth 有無 | テーマ内広がりが効くか |
| Crowding 控除有無 | 過熱回避が効くか |
| セクター中立 | セクター偏りを除いても有効か |
| 実行ラグ | ニュース時刻から翌日以降で検証 |
| コスト後 | turnover と売買コストを控除 |

今回の最小戦略は、MSCI の thematic rotation 的な考え方に近いです。MSCI World Thematic Rotation Select Index は、テーマごとの media sentiment score を使ってテーマを選択するインデックスで、上位テーマを選んで配分する実務例です。

**今回の実装方針**

## 2.3 Sentence embedding / 意味類似

テーマ名だけではニュースを拾い切れません。たとえば AI テーマでも、記事中には “GPU demand”, “data center capex”, “inference workload”, “cloud infrastructure” などが出るため、テーマ説明文・構成銘柄概要・ニュース本文を embedding 化して cosine similarity で比較します。

**今回の実装方針**

- ニュース本文または headline を embedding 化する。
- テーマプロファイルも embedding 化する。
- ニュースとテーマの cosine similarity を関連度の主成分にする。
- モデルはまず `sentence-transformers` の多言語モデルまたは英語モデルを使う。

# 5. テーマプロファイル作成方法

ここが最重要です。

## 5.1 テーマプロファイルの思想

テーマプロファイルは、単なるテーマ名ではなく、次の情報を統合した**投資テーマの意味表現**です。

$$
P_k =
[
\text{theme name},
\text{theme description},
\text{catalyst keywords},
\text{positive events},
\text{negative events},
\sum_i w_{i,k} \cdot \text{company profile}_i,
\sum_i w_{i,k} \cdot \text{industry tags}_i
]
$$

ニュースとの関連度を測るため、プロファイルは以下の4つに分けて保持する。

| profile field | 内容 | 用途 |
|---|---|---|
| core_text | テーマの本質的説明 | embedding 類似度 |
| catalyst_text | ニュースで出やすい触媒語 | keyword / embedding 補助 |
| constituent_text | 構成銘柄の事業概要をウェイト加重 | 実際の保有銘柄との接続 |
| event_text | ポジティブ・ネガティブイベント | Tone 補助 |

---

## 5.2 テーマプロファイル作成ルール

### Rule 1：テーマ名だけを使わない

悪い例：

```yaml
theme_name: AI
```

良い例：

```yaml
description: >
  Companies exposed to artificial intelligence infrastructure,
  semiconductors, GPUs, data centers, cloud computing, enterprise AI software,
  and machine learning workloads.
```

---

### Rule 2：構成銘柄概要を必ず入れる

テーマの実体は、名称ではなく構成銘柄で決まる。

たとえば同じ AI テーマでも、構成銘柄が NVIDIA, AMD, TSMC 中心なら半導体テーマに近く、Microsoft, ServiceNow, Salesforce 中心ならソフトウェアテーマに近い。

したがって、

$$
\text{constituent\_text}_k
=
\sum_i w_{i,k} \cdot \text{company\_profile}_i
$$

を作る。

実装上は重みをテキストに直接掛けるのではなく、次のようにする。

```python
def build_weighted_constituent_text(constituents, max_names=20):
    parts = []
    for c in sorted(constituents, key=lambda x: x["weight"], reverse=True)[:max_names]:
        parts.append(
            f"{c['name']} ({c['weight']:.1%} weight): {c.get('company_profile', '')}"
        )
    return "\n".join(parts)
```

| theme_type | 例 | 関連度で重視するもの |

| macro_theme | 円高メリット, 金利上昇 | topic, macro keyword |
| event_theme | 自社株買い, M&A | event keyword, entity |

```python
                return c
        return None

    headline_col = pick(headline_candidates)
```

```python
from bs4 import BeautifulSoup

def strip_html(text: str) -> str:
    if not isinstance(text, str):
        return ""
    return BeautifulSoup(text, "html.parser").get_text(" ")

def normalize_whitespace(text: str) -> str:
    return re.sub(r"\s+", " ", text or "").strip()

def clean_news_text(headline: str, story_text: str) -> str:
    headline = normalize_whitespace(strip_html(headline))
    story = normalize_whitespace(strip_html(story_text))
    if story:
        return f"{headline}. {story}"
    return headline
```

```python
        batch_size=batch_size,

def add_embeddings(df: pd.DataFrame, text_col: str, model, batch_size: int = 64) -> pd.DataFrame:

    out["embedding"] = list(emb)
    return out

def cosine_matrix(a: np.ndarray, b: np.ndarray) -> np.ndarray:
    # Assumes normalized embeddings.
    return a @ b.T
```

## 7.6 `linker.py`

役割：ニュースをテーマに多ラベル帰属する。

## 関連度定義

$$
\text{Rel}_{d,k}
=
\lambda_1 \cdot \text{EmbeddingSim}_{d,k}
+
\lambda_2 \cdot \text{EntityOverlap}_{d,k}
+
\lambda_3 \cdot \text{KeywordMatch}_{d,k}
$$

初期実装では TopicMatch は入れなくてよい。LSEG Topic の返却形式が環境差を持つため、最初は後回しにする。

```python
import numpy as np
import pandas as pd

def minmax_clip(x: float, lo: float = 0.0, hi: float = 1.0) -> float:
    return max(lo, min(hi, x))

def keyword_score(text: str, keywords: list[str]) -> float:
    if not isinstance(text, str) or not keywords:
        return 0.0
    lower = text.lower()
    hits = sum(1 for kw in keywords if kw.lower() in lower)
    return minmax_clip(hits / max(3, len(keywords) ** 0.5))

def entity_overlap_score(news_rics: list[str], theme_rics: list[str]) -> float:
    if not news_rics or not theme_rics:
        return 0.0
    s_news = set(news_rics)
    s_theme = set(theme_rics)
    return 1.0 if s_news & s_theme else 0.0

def build_news_theme_links(
    news_df: pd.DataFrame,
    theme_df: pd.DataFrame,
    sim_matrix: np.ndarray,
    themes_config: dict,
    threshold: float = 0.35,
    weights: dict | None = None,
) -> pd.DataFrame:
    if weights is None:
        weights = {"embedding": 0.70, "entity": 0.15, "keyword": 0.15}

    theme_by_id = {t["theme_id"]: t for t in themes_config["themes"]}
    rows = []

    for i, news_row in news_df.reset_index(drop=True).iterrows():
        text = news_row.get("clean_text", "")
        news_rics = news_row.get("related_rics", [])
        if not isinstance(news_rics, list):
            news_rics = []

        for j, theme_row in theme_df.reset_index(drop=True).iterrows():
            theme_id = theme_row["theme_id"]
            theme = theme_by_id[theme_id]

            emb_score = float(sim_matrix[i, j])
            emb_score01 = (emb_score + 1.0) / 2.0

            ent_score = entity_overlap_score(news_rics, theme_row.get("constituent_rics", []))
            kw_score = keyword_score(text, theme.get("catalysts", []))

            rel_score = (
                weights["embedding"] * emb_score01
                + weights["entity"] * ent_score
                + weights["keyword"] * kw_score
            )

            if rel_score >= threshold:
                rows.append({
                    "story_id": news_row["story_id"],
                    "timestamp_utc": news_row["timestamp_utc"],
                    "theme_id": theme_id,
                    "rel_score": rel_score,
                    "embedding_score": emb_score01,
                    "entity_score": ent_score,
                    "keyword_score": kw_score,
                    "headline": news_row.get("headline", ""),
                })

    return pd.DataFrame(rows)
```

# 8. Buzz / Tone / Breadth の定義

## 8.1 Buzz

テーマ $k$、日付 $t$ の関連ニュース量。

$$
\text{Mentions}_{k,t}
=
\sum_{d \in t} \mathbf{1}(\text{Rel}_{d,k} \ge \theta)
$$

関連度で加重する場合は、

$$
\text{WeightedMentions}_{k,t}
=
\sum_{d \in t} \text{Rel}_{d,k}
$$

Buzz は過去平均との差。

$$
\text{Buzz}_{k,t}
=
z_{\text{cross-section}}
\left[
\log(1+\text{WeightedMentions}_{k,t})
-
\log(1+\overline{\text{WeightedMentions}}_{k,t-20:t-1})
\right]
$$

初期実装では 20営業日平均。

---

## 8.2 Tone

最初は FinBERT を導入してもよいが、最小実装では辞書ベースの簡易 tone でもよい。ただし実運用では FinBERT 推奨。

最小実装の tone：

```python
POSITIVE_WORDS = [
    "beat", "beats", "raise", "raises", "raised", "upgrade", "upgraded",
    "growth", "strong", "record", "surge", "buyback", "dividend increase",
    "approval", "expansion", "profit rises"
]

NEGATIVE_WORDS = [
    "miss", "misses", "cut", "cuts", "downgrade", "downgraded",
    "weak", "decline", "fall", "probe", "investigation", "lawsuit",
    "delay", "loss", "warning", "guidance cut"
]
```

$$
\text{Tone}_{d}
=
\frac{N_{pos,d} - N_{neg,d}}{N_{pos,d}+N_{neg,d}+1}
$$

テーマ別 tone：

$$
\text{Tone}_{k,t}
=
\frac{
\sum_{d \in t} \text{Rel}_{d,k} \cdot \text{Tone}_{d}
}{
\sum_{d \in t} \text{Rel}_{d,k} + \epsilon
}
$$

---

## 8.3 Breadth

テーマ内でニュースが広がっているかを測る。

初期実装では、構成銘柄 RIC がニュースに含まれるかで測る。

$$
\text{Breadth}_{k,t}
=
\sum_{i \in k} w_{i,k} \cdot \mathbf{1}(\text{NewsMention}_{i,t}=1)
$$

RIC が取れない場合は、企業名の文字列一致で代替する。

```python
def compute_breadth_for_theme_day(news_rows, constituents):
    mentioned = set()
    for _, row in news_rows.iterrows():
        rics = row.get("related_rics", [])
        if isinstance(rics, list):
            mentioned.update(rics)

    breadth = 0.0
    for c in constituents:
        if c.get("ric") in mentioned:
            breadth += float(c.get("weight", 0.0))
    return breadth
```

# 9. `features.py`

```python
import numpy as np
import pandas as pd

POSITIVE_WORDS = [
    "beat", "beats", "raise", "raises", "raised", "upgrade", "upgraded",
    "growth", "strong", "record", "surge", "buyback", "dividend increase",
    "approval", "expansion", "profit rises"
]

NEGATIVE_WORDS = [
    "miss", "misses", "cut", "cuts", "downgrade", "downgraded",
    "weak", "decline", "fall", "probe", "investigation", "lawsuit",
    "delay", "loss", "warning", "guidance cut"
]

def simple_tone(text: str) -> float:
    if not isinstance(text, str):
        return 0.0
    lower = text.lower()
    pos = sum(1 for w in POSITIVE_WORDS if w in lower)
    neg = sum(1 for w in NEGATIVE_WORDS if w in lower)
    return (pos - neg) / (pos + neg + 1.0)

def add_tone(news_df: pd.DataFrame) -> pd.DataFrame:
    out = news_df.copy()
    out["tone"] = out["clean_text"].apply(simple_tone)
    return out

def cross_sectional_zscore(df: pd.DataFrame, value_col: str, date_col: str = "date") -> pd.Series:
    def z(x):
        std = x.std(ddof=0)
        if std == 0 or np.isnan(std):
            return x * 0.0
        return (x - x.mean()) / std
    return df.groupby(date_col)[value_col].transform(z)

def build_daily_theme_features(
    news_df: pd.DataFrame,
    links_df: pd.DataFrame,
    themes_config: dict,
    lookback_days: int = 20,
) -> pd.DataFrame:
    news_cols = ["story_id", "clean_text", "tone", "related_rics"]
    merged = links_df.merge(news_df[news_cols], on="story_id", how="left")
    merged["date"] = pd.to_datetime(merged["timestamp_utc"], utc=True).dt.date
    merged["date"] = pd.to_datetime(merged["date"])

    # Mentions and weighted mentions.
    g = merged.groupby(["date", "theme_id"])
    feat = g.agg(
        mentions=("story_id", "nunique"),
        weighted_mentions=("rel_score", "sum"),
        avg_rel_score=("rel_score", "mean"),
    ).reset_index()

    # Tone.
    merged["weighted_tone"] = merged["rel_score"] * merged["tone"]
    tone = merged.groupby(["date", "theme_id"]).agg(
        weighted_tone_sum=("weighted_tone", "sum"),
        rel_sum=("rel_score", "sum"),
    ).reset_index()
    tone["tone_raw"] = tone["weighted_tone_sum"] / (tone["rel_sum"] + 1e-12)
    feat = feat.merge(tone[["date", "theme_id", "tone_raw"]], on=["date", "theme_id"], how="left")

    # Fill full date-theme panel.
    all_dates = pd.date_range(feat["date"].min(), feat["date"].max(), freq="D")
    all_themes = [t["theme_id"] for t in themes_config["themes"]]
    panel = pd.MultiIndex.from_product([all_dates, all_themes], names=["date", "theme_id"]).to_frame(index=False)
    feat = panel.merge(feat, on=["date", "theme_id"], how="left").fillna({
        "mentions": 0,
        "weighted_mentions": 0.0,
        "avg_rel_score": 0.0,
        "tone_raw": 0.0,
    })

    # Buzz raw: log current weighted mentions minus rolling mean.
    feat = feat.sort_values(["theme_id", "date"])
    feat["wm_log"] = np.log1p(feat["weighted_mentions"])
    feat["wm_log_ma"] = (
        feat.groupby("theme_id")["wm_log"]
        .transform(lambda s: s.shift(1).rolling(lookback_days, min_periods=5).mean())
    )
    feat["buzz_raw"] = feat["wm_log"] - feat["wm_log_ma"].fillna(0.0)

    # Breadth.
    breadth_rows = []
    theme_by_id = {t["theme_id"]: t for t in themes_config["themes"]}
    for (date, theme_id), sub in merged.groupby(["date", "theme_id"]):
        constituents = theme_by_id[theme_id].get("constituents", [])
        mentioned = set()
        for rics in sub["related_rics"]:
            if isinstance(rics, list):
                mentioned.update(rics)

        breadth = 0.0
        for c in constituents:
            if c.get("ric") in mentioned:
                breadth += float(c.get("weight", 0.0))

        breadth_rows.append({
            "date": date,
            "theme_id": theme_id,
            "breadth_raw": breadth,
        })

    breadth_df = pd.DataFrame(breadth_rows)
    if not breadth_df.empty:
        feat = feat.merge(breadth_df, on=["date", "theme_id"], how="left")
    else:
        feat["breadth_raw"] = 0.0
    feat["breadth_raw"] = feat["breadth_raw"].fillna(0.0)

    # Cross-sectional standardization.
    feat["buzz"] = cross_sectional_zscore(feat, "buzz_raw")
    feat["tone"] = cross_sectional_zscore(feat, "tone_raw")
    feat["breadth"] = cross_sectional_zscore(feat, "breadth_raw")

    feat["theme_score"] = (
        0.40 * feat["buzz"]
        + 0.30 * feat["tone"]
        + 0.30 * feat["breadth"]
    )

    return feat
```

```python
import pandas as pd

def select_top_themes(
    scores: pd.DataFrame,
    n_top: int = 4,
    score_col: str = "theme_score",
) -> pd.DataFrame:
    rows = []
    for date, sub in scores.groupby("date"):
        ranked = sub.sort_values(score_col, ascending=False).head(n_top)
        for _, row in ranked.iterrows():
            rows.append({
                "date": date,
                "theme_id": row["theme_id"],
                "theme_weight": 1.0 / n_top,
                "theme_score": row[score_col],
            })
    return pd.DataFrame(rows)

def expand_to_constituent_weights(theme_weights: pd.DataFrame, themes_config: dict) -> pd.DataFrame:
    theme_by_id = {t["theme_id"]: t for t in themes_config["themes"]}
    rows = []

    for _, tw in theme_weights.iterrows():
        theme = theme_by_id[tw["theme_id"]]
        for c in theme.get("constituents", []):
            rows.append({
                "date": tw["date"],
                "theme_id": tw["theme_id"],
                "ric": c["ric"],
                "ticker": c.get("ticker", ""),
                "name": c.get("name", ""),
                "theme_weight": tw["theme_weight"],
                "constituent_weight": float(c.get("weight", 0.0)),
                "final_weight": tw["theme_weight"] * float(c.get("weight", 0.0)),
            })

    out = pd.DataFrame(rows)
    if out.empty:
        return out

    # In case same RIC appears in multiple themes, aggregate.
    out = out.groupby(["date", "ric"], as_index=False).agg(
        final_weight=("final_weight", "sum"),
        themes=("theme_id", lambda x: ",".join(sorted(set(x)))),
    )
    return out
```

# 11. バックテスト仕様

最初の検証では、テーマバスケットのリターン系列が既にある前提でもよい。

必要な入力：

```text
date
theme_id
return
```

または構成銘柄リターンからテーマリターンを作る。

$$
R_{k,t}
=
\sum_{i \in k} w_{i,k} R_{i,t}
$$

戦略リターン：

$$
R_{p,t+1}
=
\sum_k W_{k,t} R_{k,t+1}
$$

必ず1日または1リバランス分 shift する。

```python
def compute_strategy_returns(theme_weights, theme_returns):
    w = theme_weights.copy()
    r = theme_returns.copy()

    w["date"] = pd.to_datetime(w["date"])
    r["date"] = pd.to_datetime(r["date"])

    # Use today's selected weights from next return date.
    w = w.sort_values(["theme_id", "date"])
    r = r.sort_values(["theme_id", "date"])

    merged = r.merge(
        w[["date", "theme_id", "theme_weight"]],
        on=["date", "theme_id"],
        how="left",
    )

    merged["theme_weight"] = (
        merged.groupby("theme_id")["theme_weight"]
        .ffill()
        .fillna(0.0)
    )

    merged["theme_weight_lag"] = (
        merged.groupby("theme_id")["theme_weight"]
        .shift(1)
        .fillna(0.0)
    )

    merged["contribution"] = merged["theme_weight_lag"] * merged["return"]

    port = merged.groupby("date").agg(
        strategy_return=("contribution", "sum"),
        gross_exposure=("theme_weight_lag", "sum"),
    ).reset_index()

    return port
```

```markdown
## Core Method

For each theme k and news item d:

Rel_{d,k}
= 0.70 * EmbeddingSim_{d,k}
+ 0.15 * EntityOverlap_{d,k}
+ 0.15 * KeywordMatch_{d,k}

Use a configurable threshold, default 0.35.

Daily features:

WeightedMentions_{k,t} = sum_d Rel_{d,k}

Buzz_{k,t} =
cross_sectional_zscore(
  log(1 + WeightedMentions_{k,t})
  - rolling_mean_{20 days}(log(1 + WeightedMentions_{k,t}))
)

Tone_d =
(simple positive keyword count - simple negative keyword count)
/
(simple positive keyword count + simple negative keyword count + 1)

Tone_{k,t} =
sum_d Rel_{d,k} * Tone_d / (sum_d Rel_{d,k} + epsilon)

Breadth_{k,t} =
sum_i constituent_weight_{i,k} * 1{constituent i has related news on t}

ThemeScore_{k,t}
= 0.40 * Buzz_{k,t}
+ 0.30 * Tone_{k,t}
+ 0.30 * Breadth_{k,t}

Select top 4 themes by ThemeScore each rebalance date.
Assign 25% to each selected theme.
Within each theme, use existing constituent weights.
```

## ルールA：テーマの「名前」ではなく「収益ドライバー」を書く

# 3. 日本株版で最も重要な設計判断

## 3.1 ニュース取得は RIC 起点を主にする

日本株テーマでは、テーマ語だけでニュースを拾うと漏れが大きくなります。

たとえば「低PBR」「地銀」「防衛」「半導体製造装置」「電力インフラ」などは、ニュース本文にテーマ名が直接出ないことが多いです。したがって、最初の取得は以下の3本立てにします。

| 取得ルート | 内容 | 目的 |
|---|---|---|
| Route A: constituent RIC query | 構成銘柄 RIC ごとにニュース取得 | Breadth と個別銘柄材料の捕捉 |
| Route B: theme keyword query | テーマ語・政策語・業界語で取得 | テーマ全体ニュースの捕捉 |
| Route C: Japan market broad query | 日本市場全体のニュースを取得 | 未知の関連ニュースの拾い上げ |

LSEG 公式サンプルでも、RIC を用いたニュース取得例として `R:{universe} AND Language:LEN AND Source:RTRS` のような query が示されています。([GitHub](https://github.com/LSEG-API-Samples/Example.DataLibrary.Python.RequestsComparison/blob/main/src/ld_access_news.py?utm_source=chatgpt.com))  
このため、日本株版では **構成銘柄 RIC query を最も信頼する**設計にします。

---

## 3.2 日本語・英語の両対応にする

日本株ニュースには、英語 Reuters 記事、日本語記事、企業名の英語表記・日本語表記、証券コード、RIC が混在します。

したがって、テーマプロファイルは必ず以下を持ちます。

```yaml
theme_name_ja:
theme_name_en:
description_ja:
description_en:
catalysts_ja:
catalysts_en:
company_profile_ja:
company_profile_en:
```

embedding model は初期状態では以下を標準にします。

```yaml
embedding:
  model_name: sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
```

英語ニュースだけに限定できるなら `all-MiniLM-L6-v2` でもよいですが、日本株テーマでは日本語を扱える多言語モデルを標準にする方が安全です。

---

## 3.3 Breadth を最重要特徴量にする

日本株テーマバスケットでは、1銘柄のニュースがテーマ全体のニュースに見えてしまう問題が強く出ます。

たとえば、半導体テーマで東京エレクトロンだけ、地銀テーマで1行だけ、防衛テーマで三菱重工だけがニュースになっている場合、それは「テーマ全体の盛り上がり」ではなく「個別銘柄材料」の可能性があります。

したがって、日本株版の初期スコアは以下に変更します。

$$
\text{ThemeScore}_{k,t}
=
0.35 \cdot \text{Buzz}_{k,t}
+
0.25 \cdot \text{Tone}_{k,t}
+
0.40 \cdot \text{Breadth}_{k,t}
$$

前回は Buzz 0.40 / Tone 0.30 / Breadth 0.30 でしたが、日本株版では **Breadth を 0.40 に引き上げる**のがよいです。

## 4.2 テーマプロファイルの4層構造

| 層 | 内容 | 目的 |

| `core_text_ja/en` | テーマの本質説明 | embedding 類似度 |

# 8. 日本株版 Buzz / Tone / Breadth

## 8.1 Buzz

$$
\text{WeightedMentions}_{k,t}
=
\sum_{d \in t} \text{Rel}_{d,k}
$$

$$
\text{Buzz}_{k,t}
=
z_{cs}
\left[
\log(1+\text{WeightedMentions}_{k,t})
-
\overline{
\log(1+\text{WeightedMentions}_{k,t})
}_{t-20:t-1}
\right]
$$

これは前回と同じです。

ただし、日本株では記事本数が少ないテーマが多いため、rolling window は `20営業日` ではなく、まず `60 calendar days` または `40営業日` も比較すべきです。

初期値：

```yaml
features:
  buzz_lookback_days: 40
```

ただし、金融・銀行テーマでは「金利上昇」はポジティブになりやすく、グロース株や不動産ではネガティブになりやすいです。したがって、Tone は全テーマ共通ではなく、将来的にはテーマ別辞書にします。

初期実装では、以下で十分です。

$$
\text{Tone}_{d}
=
\frac{N_{pos,d} - N_{neg,d}}{N_{pos,d}+N_{neg,d}+1}
$$

$$
\text{Tone}_{k,t}
=
\frac{
\sum_d \text{Rel}_{d,k} \cdot \text{Tone}_{d}
}{
\sum_d \text{Rel}_{d,k}+\epsilon
}
$$

## 8.3 Breadth

日本株版の最重要指標です。

$$
\text{Breadth}_{k,t}
=
\sum_{i \in k} w_{i,k}
\cdot
\mathbf{1}(\text{NewsMention}_{i,t}=1)
$$

さらに、テーマによっては「銘柄数ベース」と「ウェイトベース」の両方を持ちます。

$$
\text{BreadthCount}_{k,t}
=
\frac{
\# \{ i \in k : \text{NewsMention}_{i,t}=1 \}
}{
\# \{ i \in k \}
}
$$

$$
\text{BreadthWeight}_{k,t}
=
\sum_{i \in k} w_{i,k}
\mathbf{1}(\text{NewsMention}_{i,t}=1)
$$

初期実装の ThemeScore では `BreadthWeight` を使い、レポートには両方出します。

```python
def build_theme_profile_text_jp(theme: dict) -> dict:
    catalysts_ja = "、".join(theme.get("catalysts_ja", []))
    catalysts_en = "; ".join(theme.get("catalysts_en", []))
    pos_ja = "、".join(theme.get("positive_events_ja", []))
    pos_en = "; ".join(theme.get("positive_events_en", []))
    neg_ja = "、".join(theme.get("negative_events_ja", []))
    neg_en = "; ".join(theme.get("negative_events_en", []))

    constituents = sorted(
        theme.get("constituents", []),
        key=lambda x: float(x.get("weight", 0.0)),
        reverse=True,
    )

    # 上位30銘柄または累積80%まで
    selected = []
    cum_w = 0.0
    for c in constituents:
        if len(selected) >= 30:
            break
        selected.append(c)
        cum_w += float(c.get("weight", 0.0))
        if cum_w >= 0.80:
            break

    constituent_lines_ja = []
    constituent_lines_en = []
    for c in selected:
        w = float(c.get("weight", 0.0))
        constituent_lines_ja.append(
            f"{c.get('name_ja','')}（{c.get('ticker','')}, {c.get('ric','')}, weight {w:.2%}）: "
            f"{c.get('company_profile_ja','')}"
        )
        constituent_lines_en.append(
            f"{c.get('name_en','')} ({c.get('ticker','')}, {c.get('ric','')}, weight {w:.2%}): "
            f"{c.get('company_profile_en','')}"
        )

    core_text = f"""
テーマ名: {theme.get("theme_name_ja", "")}
Theme name: {theme.get("theme_name_en", "")}
Theme type: {theme.get("theme_type", "")}

日本語説明:
{theme.get("description_ja", "")}

English description:
{theme.get("description_en", "")}
"""

    catalyst_text = f"""
日本語カタリスト:
{catalysts_ja}

English catalysts:
{catalysts_en}

日本語ポジティブイベント:
{pos_ja}

English positive events:
{pos_en}

日本語ネガティブイベント:
{neg_ja}

English negative events:
{neg_en}
"""

    constituent_text = f"""
構成銘柄概要:
{chr(10).join(constituent_lines_ja)}

Constituent profiles:
{chr(10).join(constituent_lines_en)}
"""

    profile_text = f"{core_text}\n{catalyst_text}\n{constituent_text}"

    return {
        "theme_id": theme["theme_id"],
        "theme_name_ja": theme.get("theme_name_ja", ""),
        "theme_name_en": theme.get("theme_name_en", ""),
        "theme_type": theme.get("theme_type", ""),
        "core_text": core_text.strip(),
        "catalyst_text": catalyst_text.strip(),
        "constituent_text": constituent_text.strip(),
        "profile_text": profile_text.strip(),
        "constituent_rics": [c.get("ric") for c in theme.get("constituents", [])],
        "constituent_tickers": [c.get("ticker") for c in theme.get("constituents", [])],
        "constituent_names_ja": [c.get("name_ja") for c in theme.get("constituents", [])],
        "constituent_names_en": [c.get("name_en") for c in theme.get("constituents", [])],
    }
```

## 12.2 日本株 entity linking

```python
def detect_related_entities_jp(text: str, constituents: list[dict]) -> dict:
    if not isinstance(text, str):
        text = ""

    matched_rics = []
    matched_tickers = []
    matched_names_ja = []
    matched_names_en = []

    lower = text.lower()

    for c in constituents:
        ric = c.get("ric", "")
        ticker = str(c.get("ticker", ""))
        name_ja = c.get("name_ja", "")
        name_en = c.get("name_en", "")

        hit = False

        if ticker and ticker in text:
            hit = True
        if name_ja and name_ja in text:
            hit = True
        if name_en and name_en.lower() in lower:
            hit = True

        if hit:
```

## 12.3 日本語・英語 keyword score

```python
def keyword_score_multilingual(text: str, keywords_ja: list[str], keywords_en: list[str]) -> float:
    if not isinstance(text, str):
        return 0.0

    lower = text.lower()
    hits = 0
    total = 0

    for kw in keywords_ja:
        if kw:
            total += 1
            if kw in text:
                hits += 1

    for kw in keywords_en:
        if kw:
            total += 1
            if kw.lower() in lower:
                hits += 1

    if total == 0:
        return 0.0

    # 少数ヒットでも意味を持たせるため sqrt で正規化
    return min(1.0, hits / max(3.0, total ** 0.5))
```

## 12.4 日本株版 Tone

```python
def simple_tone_multilingual(
    text: str,
    positive_ja: list[str],
    negative_ja: list[str],
    positive_en: list[str],
    negative_en: list[str],
) -> float:
    if not isinstance(text, str):
        return 0.0

    lower = text.lower()

    pos = 0
    neg = 0

    for w in positive_ja:
        if w and w in text:
            pos += 1

    for w in negative_ja:
        if w and w in text:
            neg += 1

    for w in positive_en:
        if w and w.lower() in lower:
            pos += 1

    for w in negative_en:
        if w and w.lower() in lower:
            neg += 1

    return (pos - neg) / (pos + neg + 1.0)
```

# 8. `embedding_text`

`embedding_text` は、ニュース本文との cosine similarity に直接使うテキストである。

## 8.1 作成ルール

`embedding_text` には以下を含める。

1. `theme_name_en`
2. `theme_name_ja`
3. `description_en`
4. `description_ja`
5. `profile_summary_en`
6. `profile_summary_ja`
7. `revenue_drivers_en`
8. `revenue_drivers_ja`
9. `catalysts_en`
10. `catalysts_ja`
11. 上位構成銘柄の企業説明

## 8.2 注意点

- 長すぎると embedding の焦点がぼやけるため、上位構成銘柄は原則として **累積ウェイト80%まで** または **上位30社まで**。
- 企業説明はそのまま全部貼るのではなく、必要に応じて短く要約する。
- 1つの企業だけが過度に支配しないようにする。
- ニュースに出やすい語句を適度に含める。
- 日本語と英語を両方含める。

# 11. `top_constituents`

個別企業リストから作る。

## 11.1 選定ルール

- 原則としてウェイト降順。
- 上位30社まで、または累積ウェイト80%まで。
- テーマが少数銘柄なら全銘柄。
- `weight` は元データの値を使う。
- 欠損は空文字または `null` にする。

```yaml
top_constituents:
  - company_id: "C001"
    ric: "8035.T"
    company_name_en: "Tokyo Electron"
    company_name_ja: "東京エレクトロン"
    weight: 0.12
    company_description_en: >
      Tokyo Electron is a Japanese semiconductor production equipment company
      exposed to wafer fabrication equipment demand and semiconductor capital expenditure.
    gics_sector: "Information Technology"
    gics_industry: "Semiconductors & Semiconductor Equipment"
    trbc_industry: "Semiconductor Equipment"
```

# 12. 出力品質に関する制約

## 12.1 してはいけないこと

- 入力データにない銘柄を勝手に追加しない。
- RIC を変更しない。
- ウェイトを変更しない。
- テーマIDを変更しない。
- テーマ説明を過度に美化しない。
- 投資推奨文にしない。
- 「必ず上昇する」「恩恵を受ける」と断定しない。
- ニュース分類に使えない抽象語だけにしない。
- 一般語を keyword に大量投入しない。
- 企業説明から読み取れない事業を勝手に追加しない。

## 12.2 してよいこと

- テーマ説明をニュース分類に適した形に補強する。
- 英語説明から自然な日本語表現を作る。
- 日本語説明から自然な英語表現を作る。
- 企業説明と業種から revenue driver を推定する。
- 業種情報から関連産業を集約する。
- 重要な政策・マクロ要因を、テーマ説明と企業群から合理的に推定する。
- 明らかな同義語・表記ゆれを catalyst に追加する。

# 16. 最終的な運用イメージ

この指示書で作成されたテーマプロファイルは、後続処理では次のように使います。

| プロファイル項目 | 用途 |
|---|---|
| `embedding_text` | ニュース本文との embedding 類似度 |
| `keyword_match_terms_ja/en` | キーワード一致スコア |
| `entity_linking_terms` | RIC・企業名一致 |
| `positive_events_ja/en` | Tone のポジティブ辞書 |
| `negative_events_ja/en` | Tone のネガティブ辞書 |
| `top_constituents` | Breadth 計算、説明、デバッグ |

# 4. `related_rics` がない場合の代替

## 対応A：企業名・証券コードから `matched_rics` を作る

日本株では、ニュース本文に RIC がなくても、企業名や証券コードが入っていることがあります。

したがって、`related_rics` がない場合は、テーマプロファイルの `entity_linking_terms` から企業名一致で補完します。

```python
def ensure_list(x):
    if isinstance(x, list):
        return x
    if pd.isna(x):
        return []
    if isinstance(x, str):
        if x.strip() == "":
            return []
        return [v.strip() for v in x.split(",")]
    return []

def infer_related_rics_from_text(text: str, theme_profiles: list[dict]) -> list[str]:
    if not isinstance(text, str):
        text = ""

    lower = text.lower()
    matched = set()

    for theme in theme_profiles:
        terms = theme.get("entity_linking_terms", {})

        rics = terms.get("rics", [])
        names_en = terms.get("company_names_en", [])
        names_ja = terms.get("company_names_ja", [])

        for ric, name_en, name_ja in zip(rics, names_en, names_ja):
            hit = False

            if name_ja and name_ja in text:
                hit = True

            if name_en and name_en.lower() in lower:
                hit = True

            # ticker / code が別途あるならここに追加
            if hit and ric:
                matched.add(ric)

    return sorted(matched)
```

```python
news_df["related_rics"] = news_df["related_rics"].apply(ensure_list)

mask_empty = news_df["related_rics"].apply(len).eq(0)

news_df.loc[mask_empty, "related_rics"] = news_df.loc[mask_empty, "clean_text"].apply(
    lambda x: infer_related_rics_from_text(x, theme_profiles)
)
```

## 1. まず `_safe_list()` を少しだけ強化

既存の `_safe_list()` を以下に置き換えてください。

```python
def _safe_list(x: Any) -> list[str]:
    if x is None:
        return []
    try:
        if pd.isna(x):
            return []
    except Exception:
        pass

    if isinstance(x, list):
        return [str(v).strip() for v in x if str(v).strip()]
    if isinstance(x, tuple) or isinstance(x, set):
        return [str(v).strip() for v in x if str(v).strip()]

    s = str(x).strip()
    if not s:
        return []

    # stringified list対策: "['8306.T', '8035.T']"
    if s.startswith("[") and s.endswith("]"):
        try:
            import ast
            vals = ast.literal_eval(s)
            if isinstance(vals, list):
                return [str(v).strip() for v in vals if str(v).strip()]
        except Exception:
            pass

    for sep in [",", ";", "|"]:
        if sep in s:
            vals = [v.strip() for v in s.split(sep) if v.strip()]
            if vals:
                return vals

    return [s]
```

元コードではスペース区切りも分割対象でしたが、企業名やニュース topic で不要分割が起きやすいので、**スペース分割は外す**方が安全です。

## 2. `normalize_headlines_df()` を最小修正版に置き換え

既存の関数だけを以下に置き換えてください。

```python
def normalize_headlines_df(
    df: pd.DataFrame,
    query_name: str,
    query_type: str = "unknown",
) -> pd.DataFrame:
    if df is None or len(df) == 0:
        return pd.DataFrame(columns=[
            "story_id",
            "headline",
            "timestamp_utc",
            "timestamp_jst",
            "timestamp_source",
            "source",
            "language",
            "topics",
            "related_rics",
            "related_rics_lseg",
            "query_name",
            "query_type",
            "retrieved_at_utc",
        ])

    out = df.copy()

    story_col = _pick_col(out, ["storyId", "story_id", "StoryId", "Story ID", "storyID"])
    headline_col = _pick_col(out, ["headline", "Headline", "text", "Text", "storyHeadline"])

    # versionCreated を第一候補にする
    time_col = _pick_col(out, [
        "versionCreated",
        "firstCreated",
        "created",
        "timestamp",
        "Timestamp",
        "Date",
        "date",
        "Time",
        "time",
    ])

    source_col = _pick_col(out, ["source", "Source", "provider"])
    lang_col = _pick_col(out, ["language", "Language"])
    topics_col = _pick_col(out, ["topics", "Topics", "topic", "Topic"])
    rics_col = _pick_col(out, [
        "related_rics",
        "relatedRics",
        "relatedRICs",
        "Related RICs",
        "RIC",
        "rics",
        "Rics",
        "subjects",
        "Subjects",
    ])

    if story_col is None:
        raise ValueError(f"Cannot find story id column. columns={list(out.columns)}")

    norm = pd.DataFrame()
    norm["story_id"] = out[story_col].astype(str).str.strip()
    norm["headline"] = out[headline_col].astype(str) if headline_col else ""

    if time_col:
        norm["timestamp_utc"] = pd.to_datetime(out[time_col], errors="coerce", utc=True)
        norm["timestamp_source"] = time_col
    else:
        norm["timestamp_utc"] = pd.NaT
        norm["timestamp_source"] = "missing"

    norm["retrieved_at_utc"] = pd.Timestamp.now(tz="UTC")

    # 小規模検証用 fallback
    mask_nat = norm["timestamp_utc"].isna()
    norm.loc[mask_nat, "timestamp_utc"] = norm.loc[mask_nat, "retrieved_at_utc"]
    norm.loc[mask_nat, "timestamp_source"] = "retrieved_at_fallback"

    norm["timestamp_jst"] = norm["timestamp_utc"].dt.tz_convert("Asia/Tokyo")

    norm["source"] = out[source_col].astype(str) if source_col else ""
    norm["language"] = out[lang_col].astype(str) if lang_col else ""
    norm["topics"] = out[topics_col].apply(_safe_list) if topics_col else [[] for _ in range(len(out))]

    norm["related_rics_lseg"] = out[rics_col].apply(_safe_list) if rics_col else [[] for _ in range(len(out))]
    norm["related_rics"] = norm["related_rics_lseg"]

    norm["query_name"] = query_name
    norm["query_type"] = query_type

    norm = norm[
        norm["story_id"].notna()
        & norm["story_id"].astype(str).str.len().gt(0)
        & norm["story_id"].astype(str).ne("nan")
    ].copy()

    return norm.drop_duplicates(["story_id", "query_name"]).reset_index(drop=True)
```
