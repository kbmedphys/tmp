## 重大な修正点

### 1. Mosaic + bootstrap が論文手法としては不十分

最大の論点はここです。Notebookでは、残差を時間ブロック単位でシャッフルし、擬似リターンを作り、再度クロスセクション回帰してAPWCを再計算しています。これは「blocked residual permutation + re-residualization」としては妥当な近似ですが、論文の Mosaic + bootstrap そのものとは言い切れません。

論文では、データを時間方向・銘柄方向の非重複パネルに分割し、各パネル内で回帰残差化を行い、時間ブロックと銘柄パネルを使って相関構造を壊します。さらに、デフォルトでは10観測日バッチに分け、各グループがファクター数の少なくとも5倍の銘柄数を持つように銘柄グループを作る、とされています。`Thematic Investing A Risk-based Perspective.pdf`

Notebookの現状では、各tickerのブロック順序を独立に入れ替える実装に近く、Mosaicの「銘柄パネル × 時間バッチ」構造を十分に再現していません。また、ブートストラップごとにフルユニバース回帰を再実行しており、論文が強調する「パネル内での残差化」「パネル間の回帰アーティファクト抑制」という要件が弱いです。このままだと、z-scoreの水準が過大または過小になり得ます。

**修正方針:**  
`mosaic_residualize_by_tiles()` のような関数を作り、時間バッチ、銘柄パネル、パネル内回帰、ブロック単位 permutation、再残差化を明示的に分離してください。少なくとも、現在の関数名を `mosaic_bootstrap_apwc` とするのは強すぎるため、実装が完全化するまでは `blocked_bootstrap_apwc` 程度に改名した方が安全です。

### 2. 論文のリリース日前後分析を再現していない

論文の Exhibit 7 は、Goldman Sachs のテーマバスケットについて、**リリース前60暦日**と**リリース後60暦日**のAPWC z-scoreを比較しています。論文では145本中61本、42%がリリース前に、60本、41%がリリース後に z-score 2以上であり、pre/post z-score相関も49%と報告されています。`Thematic Investing A Risk-based Perspective.pdf` また、リリース日前後60暦日データを用いて各バスケットごとにbootstrapを行ったと説明されています。`Thematic Investing A Risk-based Perspective.pdf`

一方、Notebookの `choose_analysis_dates()` は、原則としてリリース日から60営業日後以降の分析日を選び、そこから過去60営業日と将来60営業日を使います。これは**公開後に投資可能なライブ運用シグナル**としては保守的で良い設計ですが、論文の「リリース前後の有意性と持続性」を再現するものではありません。

**修正方針:**  
分析モードを2つに分けるべきです。

`paper_release_prepost` モードでは、リリース日を基準に、リリース前60暦日、リリース後60暦日のAPWC z-scoreを計算します。ただし、公開前バスケット構成を使うため、これはex post検証であり投資可能性とは切り分けて表示します。

`live_post_release` モードでは、現行Notebookのように公開済み構成だけを使い、`t`までの情報で `t+1` 以降を評価します。

この区別を明示しないと、「論文再現」と「実運用可能性」が混ざります。

### 3. bootstrap回数が少なく、z-score閾値2の判定が不安定

Notebookの設定では synthetic で40回、real で200回程度のbootstrapになっています。z-score自体はbootstrap分布の平均・標準偏差から出しているため、必ずしもtail probabilityを直接推定しているわけではありませんが、z≥2の境界判定に使うには200回はかなり粗いです。

論文では z-score 2以上を統計的有意性の目安として使っています。`review.md` その判定を実データで投資判断に使うなら、少なくとも研究モードでは1,000回以上、できれば結果の安定性をbootstrap seed別に確認すべきです。

**修正方針:**  
Notebookには `smoke`, `research`, `production` の3モードを用意し、smokeでは少数回、researchでは1,000回以上、productionでは並列化・キャッシュ込みでさらに多く回せるようにしてください。z-scoreだけでなく、bootstrap平均、標準偏差、Monte Carlo誤差、観測APWC、bootstrap分布の分位点も保存すべきです。

### 4. デフォルト実行が重すぎる

Notebookをスクリプト化してデフォルト設定で実行したところ、300秒でタイムアウトしました。bootstrap回数と分析日数を一時的に大幅に減らすと完走したため、主因はbootstrap内で毎回フルユニバースの日次回帰を繰り返していることです。

この性能では、日本株ユニバース、複数テーマ、複数分析日、1,000 bootstrap reps の本番検証は現実的ではありません。

**修正方針:**

### 6. synthetic data のfactor設計にrank deficiencyがある

synthetic dataでは `market` exposure が全銘柄1.0に設定され、回帰時にinterceptも追加されています。この場合、interceptとmarket factorが完全共線になります。`np.linalg.lstsq` は解を返しますが、factor returnの解釈は不定です。残差は一応得られますが、factor returnや診断結果の意味は弱まります。

**修正方針:**  
synthetic dataでは、interceptを入れるならmarket factorを外す、market factorを入れるならinterceptを外す、または制約付き回帰を使うべきです。実データでも同様に、国ファクター・市場ファクター・interceptの扱いは明示的に制御してください。

### 7. persistence regression の標準誤差が甘い

論文の Exhibit 9/10 は、過去60日残差リターンと将来60日残差リターンの関係を、APWC z-scoreが有意な群と非有意な群に分けて検証しています。論文では、z≥2群の係数0.126、t値2.11、z<2群の係数0.028、t値0.47が報告されています。`Thematic Investing A Risk-based Perspective.pdf`

Notebookでは単純OLSで同様の回帰をしていますが、同じテーマが複数回登場する、分析窓が重なる、テーマ間で銘柄が重複する、といった依存構造があります。単純OLSの標準誤差では有意性を過大評価する可能性があります。

**修正方針:**  
最低限、theme_idクラスタ、日付クラスタ、またはtwo-way clustered SEを追加してください。サンプルが少ない場合は、テーマ単位bootstrapまたはblock bootstrapを併用すべきです。

### 8. weekly strategy セクションは論文再現ではない

Notebook後半の weekly APWC rolling-Z gated momentum strategy は探索としては面白いですが、論文の主分析とは別物です。論文のcoherent theme判定はMosaic + bootstrap z-scoreに基づき、リターン持続性も残差リターンで検証しています。一方、Notebookのweekly strategyは、APWCの履歴rolling z-scoreを使い、閾値も0で、銘柄またはテーマのtotal momentumを使っています。

これは「APWCでゲートしたテーマ・モメンタム戦略」の探索であり、論文の z≥2 coherent theme シグナルとは区別すべきです。

**修正方針:**

ただし、`z >= 2` と `z < 2` で分ける。

### RRG追加

$$
future\_resid
=
\beta_0
+
\beta_1 past\_resid
+
\beta_2 x
+
\beta_3 y
+
\beta_4 \mathbf{1}\{z \ge 2\}
+
\beta_5 \mathbf{1}\{z \ge 2\} \cdot y
+
\epsilon
$$

ここで $x$ がRS-Ratio、$y$ がRS-Momentumです。

実務的には、次の比較が重要です。

| 比較 | 見たいこと |
|---|---|
| APWC gateのみ | 論文型シグナルの基本性能 |
| APWC gate + past residual return | 現在の8番シグナル |
| APWC gate + RRG quadrant | RRGでentry/exitが改善するか |
| APWC gate + RRG score | ランキング・ウェイト付けが改善するか |
| RRGのみ | APWC gateなしでも効くか。ただし本命ではない |
| total return RRG | 残差RRGとの差。ファクター混入確認用 |

## 結論

最も推奨する設計は、次です。

$$
\boxed{
\text{APWC Mosaic z-score } \ge 2
\text{ をcoherence gateにし、Residual RRGで方向・タイミング・順位を決める}
}
$$

具体的には、

$$
score_{B,t}
=
G(z_{B,t})
\cdot
\left[
\alpha \cdot
\frac{RSRatio_{B,t}-100}{10}
+
\beta \cdot
\frac{RSMomentum_{B,t}-100}{10}
\right]
$$

を作り、long-onlyなら正の上位だけ、long-shortなら正をlong・負をunderweight/shortにします。

RRGは「テーマが上向きか下向きか」を見ます。一方、APWC z-scoreは「そのテーマがそもそもcoherent themeとして成立しているか」を見ます。最終目標が `APWC z-score > 2` のシグナル応用であれば、RRGは主役ではなく、**coherent theme signalの実運用化を滑らかにするtiming/ranking overlay**として使うのがよいです。

## 1. 方法A：weekly rolling-Z gate + Return RRG

これは一番実装しやすく、戦略としても説明しやすい方式です。

まず週次リバランス日 $t$ で、テーマ $B$ のAPWCを過去60営業日などで計算します。

$$
APWC_{B,t}
$$

次に、そのテーマ自身の過去APWC履歴からrolling-Zを作ります。

$$
Z^{roll}_{B,t}
=
\frac{
APWC_{B,t}
-
\mu(APWC_{B,t-K:t-1})
}{
\sigma(APWC_{B,t-K:t-1})
}
$$

ここで重要なのは、平均・標準偏差をできれば **$t-1$ まで** で計算することです。$APWC_t$ をrolling平均に含めても将来情報ではありませんが、シグナルの解釈が少し鈍るため、運用シグナルとしてはcurrent observationを除外する方が綺麗です。

そのうえで、例えば次のように使います。

$$
\mathbf{1}\{Z^{roll}_{B,t} > \theta\}
$$

をcoherence gateにします。閾値は、初期値としては `0`, `0.5`, `1.0` あたりを比較するのがよいです。

その後、Return RRGを計算します。

$$
RS_{B,t}
=
\sum_{s=t-L+1}^{t}
\left(
r^{res}_{B,s}
-
r^{res}_{bench,s}
\right)
$$

$$
RSRatio_{B,t}
=
100 + 10 \cdot Z(RS_{B,t})
$$

$$
RSMomentum_{B,t}
=
100 + 10 \cdot Z(RS_{B,t} - RS_{B,t-k})
$$

そして、

| 条件 | アクション |
|---|---|
| `rolling_Z > threshold` and `Leading` | long |
| `rolling_Z > threshold` and `Improving → Leading` | new long |
| `rolling_Z > threshold` and `Weakening` | reduce / hold |
| `rolling_Z > threshold` and `Lagging` | underweight / short |
| `rolling_Z <= threshold` | no position |

という形にします。

これは、現在の weekly APWC rolling-Z momentum strategy を自然に拡張した形です。

## 2. 方法B：weekly rolling-Z 自体を Coherence RRG 化する

もう一つの方法は、APWC rolling-ZそのものをRRG化することです。

この場合、横軸をAPWC rolling-Zの水準、縦軸をAPWC rolling-Zの変化にします。

$$
X^{coh}_{B,t}
=
Z^{roll}_{B,t}
$$

$$
Y^{coh}_{B,t}
=
Z^{roll}_{B,t}
-
Z^{roll}_{B,t-k}
$$

またはノイズを抑えるなら、

$$
Y^{coh}_{B,t}
=
EMA(Z^{roll}_{B,t}, h_1)
-
EMA(Z^{roll}_{B,t}, h_2)
$$

のような短期EMAと長期EMAの差でもよいです。

この場合の象限は次です。

| 状態 | 条件 | 解釈 |
|---|---|---|
| Coherence Leading | `rolling_Z > 0`, `Δrolling_Z > 0` | APWCが過去比で高く、さらに上昇中 |
| Coherence Weakening | `rolling_Z > 0`, `Δrolling_Z < 0` | APWCはまだ高いが低下中 |
| Coherence Lagging | `rolling_Z < 0`, `Δrolling_Z < 0` | APWCが低く、さらに低下中 |
| Coherence Improving | `rolling_Z < 0`, `Δrolling_Z > 0` | APWCは低いが改善中 |

この方式は、**テーマのcoherence lifecycle管理**に向いています。

例えば、

```text
Coherence Improving -> Coherence Leading
```

になったテーマだけを、return RRGのlong候補に入れる、という使い方ができます。

## 3. 方法C：weekly rolling-Z と Return RRG の二層モデル

実務的には、これが一番強い設計です。

第一層で、weekly APWC rolling-Zからcoherence weightを作ります。

$$
G^{roll}_{B,t}
=
\mathrm{clip}
\left(
\frac{
Z^{roll}_{B,t} - \theta_{low}
}{
\theta_{high} - \theta_{low}
},
0,
1
\right)
$$

例えば、

$$
\theta_{low}=0,\quad \theta_{high}=1.5
$$

とします。

第二層で、Return RRG scoreを作ります。

$$
RRGScore_{B,t}
=
\alpha \cdot x_{B,t}
+
\beta \cdot y_{B,t}
$$

ここで、

$$
x_{B,t}
=
\frac{RSRatio_{B,t}-100}{10}
$$

$$
y_{B,t}
=
\frac{RSMomentum_{B,t}-100}{10}
$$

です。

最終スコアは、

$$
Score_{B,t}
=
G^{roll}_{B,t}
\cdot
RRGScore_{B,t}
$$

です。

long-onlyなら `Score > 0` の上位テーマを買い、long-shortなら正スコアをlong、負スコアをunderweight/shortにします。

## 6. rolling-Z RRG を使う場合の注意点

最大の注意点は、**weekly rolling-Zは統計的有意性ではない**ことです。

`rolling_Z > 0` は、

```text
このテーマのAPWCが、このテーマ自身の過去平均より高い
```

という意味です。

一方、`Mosaic z_score > 2` は、

```text
このテーマのAPWCが、無相関帰無分布対比で有意に高い
```

という意味です。

したがって、rolling-Z RRGを使う場合、レポートや戦略名では以下のように分けるべきです。

```text
coherent theme signal:
    Mosaic + bootstrap z_score >= 2

APWC regime / timing signal:
    weekly APWC rolling-Z and its RRG state
```

混同すると、rolling-Z戦略が論文準拠のcoherent theme戦略であるかのように見えてしまいます。

| 分析 | 目的 |
|---|---|
| rolling-Z gateあり/なし比較 | APWC regime filterに付加価値があるか |
| residual RRG vs total RRG | ファクター混入ではなく残差成分で効いているか |
| Mosaic z-score gateとの併用 | 論文型coherent theme判定が性能を改善するか |
| quadrant別 forward return | Leading / Improving / Weakening / Lagging の順序性確認 |
| transition別 forward return | Improving→Leadingがentryとして有効か |
| threshold sweep | `rolling_Z > 0, 0.5, 1.0, 1.5` の安定性確認 |
| top-N sweep | 上位テーマ数への感応度確認 |
| turnover / cost stress | 週次運用として実行可能か |
| placebo APWC gate | rolling-Z gateが偶然でないか |

特に重要なのは、`Leading`象限だけでなく、**遷移**を見ることです。

```text
Improving -> Leading
```

が将来リターンに対して最も強いなら、RRGはentry timingとして有効です。

```text
Leading -> Weakening
```

でリターンが鈍るなら、reduce ruleとして使えます。
